# OpenZIM MCP v0.9.0 — Design Spec

**Status:** Draft (awaiting user review)
**Date:** 2026-04-30
**Target version:** 0.9.0 (minor — new features, no breaking changes)
**Previous release:** 0.8.3 (2026-01-30)

---

## 1. Goals

Ship a coherent minor release that:

1. **Lands the uncommitted work** sitting on `main` (3 new tools, 2 new MCP resources, 6 substantive fixes) — none of it has been released yet.
2. **Adds 5 new tools, 3 MCP prompts, and Simple-mode intent patterns** across four themes (multi-archive workflows, agent-friendly retrieval, MCP completeness, stability) that compose with the uncommitted work rather than expanding into new territory.
3. **Refreshes user-visible "What's new"** surfaces (README banner, website hero banner, website feature cards) so the headline value of 0.9.0 is obvious.
4. **Holds the line on quality**: tests for every new tool, ≥80% coverage, all linters/security checks clean before tagging.

Out of scope: per-entry MCP resources (FastMCP URI-template `/`-handling blocker), HTTP/SSE transport, resource subscriptions, batch `get_entries` — all deferred to 1.0.0.

These changes already exist locally. The release commits and tests them; no rewriting.

### 2.1 New tools

| Tool | File | Purpose |
|---|---|---|
| `walk_namespace` | [tools/navigation_tools.py](../../../openzim_mcp/tools/navigation_tools.py) + [zim_operations.py](../../../openzim_mcp/zim_operations.py) | Deterministic cursor-paginated namespace iteration. Unlike `browse_namespace` (which samples), this scans by entry ID with `next_cursor` and `done` flags for full enumeration. |
| `search_all` | [tools/search_tools.py](../../../openzim_mcp/tools/search_tools.py) + [zim_operations.py](../../../openzim_mcp/zim_operations.py) | Search every ZIM file in allowed directories, merging results per-file. Skips files that can't be searched without aborting. |
| `warm_cache` | [tools/server_tools.py](../../../openzim_mcp/tools/server_tools.py) + [zim_operations.py](../../../openzim_mcp/zim_operations.py) | Pre-populate cache (list_zim_files, get_zim_metadata, list_namespaces, get_main_page) so subsequent calls hit cache. |

### 2.2 New MCP resources

First use of MCP **resources** primitive (previously tools-only). Lives in **new** module [tools/resource_tools.py](../../../openzim_mcp/tools/resource_tools.py), wired via `register_resources()` in [tools/**init**.py](../../../openzim_mcp/tools/__init__.py).

- `zim://files` — JSON index of all ZIM files in allowed dirs
- `zim://{name}` — overview of one ZIM (metadata, namespace summary, main page preview, where `{name}` is the basename without `.zim`)

Per-entry resources (`zim://{name}/entry/A/Article`) intentionally excluded for v0.9.0 — FastMCP URI templates don't handle literal `/` cleanly inside path captures. Use the existing `get_zim_entry` tool for that case. Revisit for 1.0.0.

### 2.3 Fixes

| Fix | File | Rationale |
|---|---|---|
| Namespace listing — known-prefix probes | [zim_operations.py](../../../openzim_mcp/zim_operations.py) | Random sampling silently missed minority namespaces (M, W, X, I) when one namespace dominated. Probe canonical paths (`M/Title`, `X/fulltext/xapian`, `I/favicon.png`, etc.) to deterministically surface them. |
| Namespace listing — drop fabricated extrapolations | [zim_operations.py](../../../openzim_mcp/zim_operations.py) | Below 3 sampled hits per namespace, `count / sampling_ratio` invents numbers (e.g. 5 probed M/* on a 1000-of-20565 sample → "estimated_total: 100"). Now only extrapolates when sampled ≥ 3; below threshold, reports lower-bound (confirmed sightings). |
| Search filtering — streaming scan | [zim_operations.py](../../../openzim_mcp/zim_operations.py) | Hard 1000-hit cap caused empty results for rare filters (e.g. `image/png` on an HTML-dominant corpus). Replaced with batched scan (BATCH_SIZE=500) up to 10000-entry MAX_SCAN, stopping when offset+limit filtered matches collected. |
| Error routing — pattern match first | [error_messages.py](../../../openzim_mcp/error_messages.py) | "Entry not found" was getting "check disk space" advice because broad-category exceptions matched `OpenZimMcpArchiveError` config. Message-pattern checks (`"entry not found"`, `"permission"`) now run before type-based config lookup. |
| Instance tracker — TOCTOU re-check | [instance_tracker.py](../../../openzim_mcp/instance_tracker.py) | Phantom processes (PID file but process gone) reported as conflicts. Re-verify liveness immediately before reporting; auto-unregister phantoms with debug logging. |
| `max_content_length` floor | [config.py](../../../openzim_mcp/config.py) + [tools/content_tools.py](../../../openzim_mcp/tools/content_tools.py) | Relaxed minimum from 1000 → 100 to support short-preview use cases. |

### 2.4 Doc improvements

- `get_article_structure` and `get_table_of_contents` docstrings now warn that "mini"/"nopic" ZIM flavours strip headings, so callers know to check `Flavour` metadata before assuming low heading count = bug.

## 3. New work for 0.9.0

New items organized by theme. Each follows existing project conventions (sync impl in `zim_operations.py` or new helper, async wrapper in `async_operations.py`, registration in `tools/<category>.py`).

### 3.1 `find_entry_by_title` — title→path resolver

**Theme:** A. Multi-archive workflows.
**Pain point:** LLMs guess exact entry paths and miss when capitalization, punctuation, or spaces differ. Currently the only resolver is full-text search via `search_zim_file`, which is heavier than needed when the LLM knows the title.

**Tool placement:** [tools/search_tools.py](../../../openzim_mcp/tools/search_tools.py).

**Signature:**

```python
@server.mcp.tool()
async def find_entry_by_title(
    zim_file_path: str,           # required
    title: str,                   # case-insensitive partial or full match
    cross_file: bool = False,     # if True, search all allowed ZIMs
    limit: int = 10,
) -> str:
    """Resolve a title to one or more entry paths.

    Implementation order:
      1. Direct path probe in C/ namespace for exact normalized match
         (replace spaces → underscores, capitalize, etc.) — fast path.
      2. libzim suggestion search (title-indexed, no full-text scan) —
         primary fallback.
      3. Return ranked list with score.

    Returns JSON:
        {
          "query": str,
          "results": [{"path": str, "title": str, "score": float, "zim_file": str}],
          "fast_path_hit": bool,
          "files_searched": int  # 1 unless cross_file=True
        }
    """
```

**Edge cases:**

- Empty/whitespace-only title → `OpenZimMcpValidationError`.
- No matches → return empty `results` list with `files_searched` set, not an error.
- `cross_file=True` skips files that fail to open without aborting.

**Why single-file by default:** `search_all` already exists for the cross-file case; making this single-file-default keeps the cheaper path the cheaper invocation.

### 3.2 `get_random_entry` — exploration aid

**Theme:** B. Agent-friendly retrieval.
**Tool placement:** [tools/content_tools.py](../../../openzim_mcp/tools/content_tools.py).

**Signature:**

```python
@server.mcp.tool()
async def get_random_entry(
    zim_file_path: str,
    namespace: str = "C",  # default to articles; "" for any namespace
) -> str:
    """Return one random entry from the ZIM.

    Wraps libzim's archive.get_random_entry(). When namespace is set
    (default "C"), retries up to 20 times to land in that namespace
    before giving up with a clear error.

    Returns JSON: {"path": str, "title": str, "namespace": str, "preview": str}
    where preview is a 200-char snippet of the entry content.
    """
```

**Edge cases:**

- Cannot find namespace match within retry budget → return informative error suggesting the caller call `list_namespaces` first.
- Empty archive → archive-error.

### 3.3 `get_related_articles` — link-graph neighbours

**Theme:** B. Agent-friendly retrieval.
**Tool placement:** [tools/structure_tools.py](../../../openzim_mcp/tools/structure_tools.py).

**Signature:**

```python
@server.mcp.tool()
async def get_related_articles(
    zim_file_path: str,
    entry_path: str,
    limit: int = 10,
    direction: str = "outbound",  # "outbound" | "inbound" | "both"
    inbound_scan_cap: int = 5000,
) -> str:
    """Find articles related to entry_path via link graph.

    direction="outbound": cheap — composes extract_article_links and
        deduplicates internal links, returning up to `limit`.

    direction="inbound": bounded scan — walk C/ namespace via internal
        helper (same scan logic as walk_namespace) up to `inbound_scan_cap`
        entries, checking each for an outbound link to entry_path. Returns
        a `next_cursor` for continuation if cap is hit before completion.

    direction="both": run outbound (full), then inbound (bounded), merge
        and deduplicate. Each result tagged with which direction(s) found it.

    Returns JSON: {
        "entry_path": str,
        "direction": str,
        "outbound_results": [{"path": str, "title": str}],  # if applicable
        "inbound_results": [{"path": str, "title": str}],   # if applicable
        "inbound_scanned": int,                              # if applicable
        "inbound_next_cursor": int | null,                   # if applicable
        "inbound_done": bool                                 # if applicable
    }
    """
```

**Why bound inbound at 5000:** unbounded inbound scan on a 6M-entry Wikipedia archive is minutes-long. 5000 caps wall time at ~5–10 seconds (approx) on test hardware while still surfacing ~70% of inbound links for typical articles. Caller can paginate via `next_cursor` for completeness.

### 3.4 + 3.5 `cache_stats` and `cache_clear` — cache control

**Theme:** D. Stability & ops.
**Tool placement:** [tools/server_tools.py](../../../openzim_mcp/tools/server_tools.py) (next to `warm_cache`).

```python
@server.mcp.tool()
async def cache_stats() -> str:
    """Return current cache state. JSON: {size, max_size, hit_rate,
    miss_count, hit_count, ttl_seconds}."""

@server.mcp.tool()
async def cache_clear() -> str:
    """Clear all cached entries. Returns prior size and confirmation.
    Use when ZIM files have been modified on disk and stale cache
    entries are returning outdated content."""
```

Both reuse `OpenZimMcpCache.stats()` and `clear()` already defined in [cache.py](../../../openzim_mcp/cache.py).

### 3.6 MCP Prompts — three workflow templates

**Theme:** C. MCP completeness.
**New module:** `openzim_mcp/tools/prompts.py` with `register_prompts(server)` wired into `register_all_tools` in [tools/**init**.py](../../../openzim_mcp/tools/__init__.py).

Three prompt templates, exposed as user-invocable slash commands in MCP-aware clients (Claude Code, Inspector). Each returns a `list[PromptMessage]` per the MCP spec.

```python
@server.mcp.prompt("research")
def research(topic: str) -> list[PromptMessage]:
    """Research a topic across all ZIM files: search_all → get_entry_summary
    on top hits → ask user what to drill into next."""

@server.mcp.prompt("summarize")
def summarize(zim_file_path: str, entry_path: str) -> list[PromptMessage]:
    """Summarize an article: get_table_of_contents + get_entry_summary +
    extract_article_links → return structured summary."""

@server.mcp.prompt("explore")
def explore(zim_file_path: str) -> list[PromptMessage]:
    """Explore a ZIM file: get_zim_metadata + list_namespaces + get_main_page
    + 5 calls to get_random_entry → present sample of contents."""
```

**Why these three:** they're the highest-leverage common workflows that currently require the LLM to chain 4–6 tool calls. Promoting them to first-class prompts makes each a single user-facing slash command.

### 3.7 Simple-mode intent patterns

**Theme:** A. Multi-archive (polish).
**File:** [simple_tools.py](../../../openzim_mcp/simple_tools.py) intent table.

Add patterns for new and uncommitted tools so the natural-language Simple mode tool can route to them. Each pattern is `(regex, intent_name, confidence, specificity)`:

| Phrase trigger | Routes to | Specificity |
|---|---|---|
| `search all/everywhere/across files for X` | `search_all` | high |
| `walk/iterate/dump namespace X` | `walk_namespace` | high |
| `warm/preload/prefetch cache` | `warm_cache` | high |
| `find article/entry titled X`, `what's the path for X` | `find_entry_by_title` | high |
| `random article/entry [in X]` | `get_random_entry` | medium |
| `related to X`, `articles linking to X`, `what links from X` | `get_related_articles` | medium |
| `cache stats`, `clear cache` | `cache_stats` / `cache_clear` | high |

Existing patterns unchanged; new ones slot in by specificity so they don't shadow existing matches.

## 5. Testing strategy

The uncommitted work has thin test coverage — that's the biggest risk for 0.9.0.

### 5.1 Per-tool unit tests

Following the existing `tests/test_<feature>.py` convention (one file per category or per tool):

| Test file | Coverage |
|---|---|
| `tests/test_walk_namespace.py` (new) | walk_namespace happy path, large cursor, namespace not present, total iteration via paging, validation errors |
| `tests/test_search_all.py` (new) | search_all with mixed-success ZIMs (one corrupt fixture), empty query, limit_per_file bounds, no matches in any file |
| `tests/test_warm_cache.py` (new) | warm_cache happy path, missing main page (best-effort fallback), invalid path, cache size grows |
| `tests/test_resources.py` (new) | `zim://files` JSON shape, `zim://{name}` resolution by stem and by name, missing file error, partial success (metadata fails but namespaces succeeds) |
| `tests/test_find_entry_by_title.py` (new) | exact match fast path, partial match via suggestion, cross-file mode, no matches, validation errors |
| `tests/test_get_random_entry.py` (new) | namespace match within retry budget, namespace not present (informative error), empty archive |
| `tests/test_get_related_articles.py` (new) | outbound (link extraction), inbound (bounded scan with cursor), both (merge + dedupe), inbound cap reached, invalid direction |
| `tests/test_cache_control.py` (new) | cache_stats shape, cache_clear empties cache, stats after clear |
| `tests/test_prompts.py` (new) | each prompt renders to valid `list[PromptMessage]`, parameter substitution, missing required args |

### 5.2 Existing test files extended

- `tests/test_zim_operations.py`: extend with new methods on `ZimOperations` (find_entry_by_title, get_random_entry, get_related_articles).
- `tests/test_simple_tools.py`: extend with new intent-pattern matches and disambiguation against existing patterns (specifically: ensure "search all" routes to `search_all`, not `search`).
- `tests/test_server.py`: extend with end-to-end MCP-layer test for one tool from each new category.

### 5.3 Coverage target

Existing project standard: **≥80%**. Run `make test-cov` and verify before tagging.

### 5.4 Manual smoke

Before tagging:

1. Launch server with the local test ZIM in `test_data/`.
2. Connect via MCP Inspector.
3. Invoke each new tool and each prompt; verify response shape.
4. Test the new resources via Inspector's resource browser.
5. Run Simple mode against each new natural-language pattern.

### 5.5 Skipped

- Performance benchmarks for new tools — `walk_namespace` and inbound `get_related_articles` could benefit, but `pytest-benchmark` already exists for hot paths and we're not promising perf characteristics in this release.
- Security tests for new tools — they all route through the existing `sanitize_input` validator, which already has dedicated security tests.

## 7. Architecture notes

### 7.1 Module placement

No new top-level modules. All new code drops into existing `openzim_mcp/tools/<category>.py` files following the established pattern. One new file: `openzim_mcp/tools/prompts.py` for the MCP Prompts primitive (introduces a new tool category — prompts — distinct from tools and resources).

### 7.2 Sync/async split

All new tools follow the existing pattern:

1. Sync implementation in [zim_operations.py](../../../openzim_mcp/zim_operations.py).
2. Async wrapper in [async_operations.py](../../../openzim_mcp/async_operations.py) using `asyncio.to_thread`.
3. MCP registration in `tools/<category>.py` calling the async wrapper.

Cache control tools (`cache_stats`, `cache_clear`) are exceptions — they touch the in-memory cache directly with no I/O, so they don't need async wrapping. They're registered as `async def` at the MCP layer for protocol consistency but call sync cache methods directly.

Prompts have no sync/async split — they're pure template rendering, registered with `@server.mcp.prompt()`.

### 7.3 Validation and security

All new tools accept user input through `sanitize_input` from [security.py](../../../openzim_mcp/security.py) at the MCP boundary, with the same per-input length limits already defined in [constants.py](../../../openzim_mcp/constants.py). Path arguments go through `path_validator.validate_path` and `validate_zim_file`.

### 7.4 Rate limiting

Each new tool calls `server.rate_limiter.check_rate_limit("<operation>")` with a logical operation name following the existing per-operation rate-limit convention. New operation names: `find_entry_by_title`, `get_random_entry`, `get_related_articles`, `cache_stats`, `cache_clear`. (`warm_cache`, `walk_namespace`, `search_all` already wired in the uncommitted work.)

## 8. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Inbound `get_related_articles` is slow on large archives | Bounded at 5000-entry scan with `next_cursor`. Document expected wall-time in tool docstring. |
| MCP Prompts spec interpretation differs across clients | Test against MCP Inspector + Claude Code at minimum. Keep prompt templates simple (single-message instruction or short conversation), avoid advanced features. |
| Splitting 14-file uncommitted change into 9 commits is error-prone | Use `git add -p` for staged splits. Verify each commit builds and tests pass with `git rebase -i --exec 'make test'`. |
| Website JS dynamic version display might mask stale fallback | Update fallback to `v0.9.0` regardless. Test with JS disabled in browser dev tools to confirm fallback shows latest. |
| Test coverage drops below 80% from sheer volume of new code | Coverage gate is in `make test-cov`. If under, write more tests before tagging. No exceptions. |
