# OpenZIM MCP v1.0.0 — Design Spec

**Status:** Draft (awaiting user review)
**Date:** 2026-04-30
**Target version:** 1.0.0 (feature release; no breaking changes. The version-number jump is symbolic — earlier brainstorming chose to treat 1.0 as "the next release after 0.9", not as an API-stability milestone, so semver-wise this is functionally a feature release.)
**Previous release:** 0.9.0 (2026-04-30)

---

## 1. Goals

Ship a coherent release that:

1. **Adds streamable-HTTP transport** as a first-class deployment shape — wired via CLI/env, secured by a bearer token, packaged as a published multi-arch Docker image, and documented for self-hosted single-user scenarios (Tailscale, VPS, homelab).
2. **Lands two of the three remaining items deferred from 0.9.0** — batch `get_zim_entries` and per-entry MCP resources — that compose naturally with HTTP.
3. **Lands resource subscriptions if they're ready**, slipping cleanly to 1.1 if not. The release is structured so subscriptions are the only thing that can move without disrupting the rest.
4. **Holds the line on quality**: tests for every new feature, ≥80% coverage, all linters/security checks clean before tagging. Stdio path remains the default and gets a full regression pass — no client should need to change configuration to upgrade.

**Explicitly not goals:**

- Multi-user / hosted deployment (per-token identity, OAuth, per-user rate limits) — out of scope; deferred to 1.x or 2.0.
- Native in-process TLS — punt to reverse proxy (caddy/nginx).
- Forced breaking changes — none. Tool names, signatures, JSON shapes preserved.
- Per-entry resource subscriptions — covered transitively by file-level subscriptions; ZIM entries are immutable per file.

## 3. Architecture

### 3.1 CLI / configuration (additive, non-breaking)

```
openzim-mcp                                       # stdio (current default)
openzim-mcp --transport http                      # HTTP, defaults to 127.0.0.1:8000
openzim-mcp --transport http --host 0.0.0.0 --port 9000
```

**New flags:** `--transport`, `--host`, `--port`. **New env vars:** `OPENZIM_MCP_TRANSPORT`, `OPENZIM_MCP_HOST`, `OPENZIM_MCP_PORT`, `OPENZIM_MCP_AUTH_TOKEN`, `OPENZIM_MCP_CORS_ORIGINS`, `OPENZIM_MCP_WATCH_INTERVAL`, `OPENZIM_MCP_SUBSCRIPTIONS_ENABLED`. **Precedence:** CLI flag > env var > default.

The user-facing `--transport` value is `http` (one word). Internally, [server.py:322](../../../openzim_mcp/server.py#L322) calls FastMCP with `"streamable-http"`. The mapping happens in [main.py](../../../openzim_mcp/main.py) at the dispatch site. SSE is intentionally not exposed (deprecated upstream).

**Default `host` is `127.0.0.1`.** Operator must explicitly choose `0.0.0.0` to expose to the network.

**Auth token is env-only.** No support for placing the token in a config file or as a CLI arg — both leak via process listings and accidental commits. Documented explicitly.

### 3.2 Config object additions

[config.py](../../../openzim_mcp/config.py) gets new optional fields:

```python
transport: Literal["stdio", "http"] = "stdio"
host: str = "127.0.0.1"
port: int = 8000  # validated 1-65535
auth_token: SecretStr | None = None  # loaded from env, never serialized
cors_origins: list[str] = []
watch_interval_seconds: int = 5  # validated 1-60
subscriptions_enabled: bool = True
```

`auth_token` uses `pydantic.SecretStr` so it doesn't leak into log lines, repr output, or error messages.

### 3.3 Instance tracker semantics

[instance_tracker.py](../../../openzim_mcp/instance_tracker.py) extends each registry record with `transport`, `host`, `port`. Conflict detection diverges by transport:

| Conflict scenario | Behavior |
|---|---|
| stdio↔stdio, same config | Existing fail-fast (with TOCTOU re-check from 0.9.0). Unchanged. |
| http↔http, same port | OS handles via `EADDRINUSE`. Tracker records both informationally; no duplicate check. |
| http↔http, different ports, same allowed-dirs | **Allowed.** Legitimate (e.g., one for read-only public, one for authenticated admin). |
| stdio + http, same config | **Allowed.** Different transports don't compete for resources. |

The `list_running_instances` debug tool surfaces all transport types in the registry.

### 3.4 Lifecycle differences

| | stdio | HTTP |
|---|---|---|
| Process lifetime | Per-client spawn (dies on stdin close) | Long-running |
| Cache scope | Per-process (rebuilt each spawn) | Shared across all sessions |
| Rate limiter | Per-process | Global (acceptable for single-user) |
| Subscriptions | Tied to process lifetime | Tied to MCP session |
| Request logging | Per-operation | Per-operation + per-request (method, session, latency) |

The shared cache in HTTP is a real win — falls out of the lifetime change with no special code.

### 3.5 Reverse-proxy assumptions

FastMCP/Starlette respects `X-Forwarded-*` headers by default. Documented to enable this only when actually behind a proxy. We don't authenticate by source IP (only by bearer token), so the impact of header spoofing is limited to log entry accuracy. No code changes; this is a deployment-doc concern.

## 4. Streamable-HTTP transport

### 4.1 Auth middleware

A single Starlette middleware checks `Authorization: Bearer <token>` against `Config.auth_token`. Comparison via `hmac.compare_digest` (timing-safe). Missing or wrong token → `401 Unauthorized` with `WWW-Authenticate: Bearer` header. Auth failures log:

- Source IP (after `X-Forwarded-For` resolution if proxy mode enabled)
- Failure type: `missing_header`, `wrong_scheme`, `invalid_token`
- Count over rolling window (for ops dashboards / log alerts)

The attempted token is **never** logged.

### 4.2 Safe-default startup check

When `transport=http`, the server runs this check before binding:

| host | auth_token | result |
|---|---|---|
| `127.0.0.1` | unset | start (localhost-only, no auth — safe) |
| `127.0.0.1` | set | start |
| any other host | unset | **refuse to start** with clear message: "Set OPENZIM_MCP_AUTH_TOKEN or bind to 127.0.0.1" |
| any other host | set | start |

This is the single most important safety property of the release. Dedicated test in `tests/test_http_auth.py`.

### 4.3 Health endpoints

Both unauthenticated, both cheap, both before auth middleware in the middleware chain:

- `GET /healthz` — `200 {"status":"ok"}`. Liveness only — proves the process is up and the event loop is running.
- `GET /readyz` — `200 {"status":"ready"}` if config is loaded and at least one allowed-dir is readable; `503 {"status":"not_ready", "reason": "..."}` otherwise. Readiness for traffic. Used by Docker `HEALTHCHECK` and reverse-proxy upstream checks.

### 4.4 CORS

`OPENZIM_MCP_CORS_ORIGINS` env var, comma-separated allow-list, default empty (no CORS headers).

- Wildcard `*` rejected at startup with explicit error (footgun prevention; operators who want it can list origins explicitly).
- `null` origin not supported.
- Preflight (`OPTIONS`) handled correctly for listed origins.

Typical operator use: set `OPENZIM_MCP_CORS_ORIGINS=http://localhost:5173` for MCP Inspector access during development.

### 4.5 Dockerfile

Multi-stage build, [Dockerfile](../../../Dockerfile) at repo root:

- **Base:** `python:3.13-slim` (final stage); separate builder stage for `uv` + deps.
- **Dep install:** `uv sync --frozen --no-dev` from the builder stage; `.venv` copied to final stage.
- **User:** non-root `appuser` (UID `10001`, GID `10001`).
- **Volume:** `/data` for ZIM files (mounted by operator).
- **Expose:** `8000`.
- **HEALTHCHECK:** `curl -f http://localhost:8000/readyz || exit 1` every 30s.
- **Entrypoint:** runs HTTP transport binding `0.0.0.0:8000` by default; `OPENZIM_MCP_*` env vars override.
- **Auth required by default in image.** Because the image entrypoint binds `0.0.0.0`, the safe-default startup check (Section 4.2) will refuse to start unless `OPENZIM_MCP_AUTH_TOKEN` is set. This is intentional — the image fails loudly with a clear error rather than starting an unauthenticated server exposed to the container network. The deployment guide opens with token generation for this reason.
- **Target final-image size:** <500 MB.
- **Multi-arch:** `linux/amd64`, `linux/arm64` via `docker buildx`.

### 4.6 CI / publishing

New workflow [.github/workflows/docker-publish.yml](../../../.github/workflows/docker-publish.yml):

- **Trigger:** tag push (`v*`) — same trigger as release-please artifacts.
- **Steps:** `docker/setup-qemu-action` (ARM emulation), `docker/setup-buildx-action`, `docker/login-action` for `ghcr.io`, `docker/build-push-action` for multi-arch build + push.
- **Tags:** `ghcr.io/cameronrye/openzim-mcp:1.0.0` and `ghcr.io/cameronrye/openzim-mcp:latest`.
- **PR builds:** verify the image builds for both arches but don't push.

### 4.7 Deployment guide

New doc [docs/HTTP_DEPLOYMENT_GUIDE.md](../../../docs/HTTP_DEPLOYMENT_GUIDE.md):

1. **Generate a token:** `python -c "import secrets; print(secrets.token_urlsafe(32))"`.
2. **Quick start with Docker:** `docker run` example with env vars and volume mount.
3. **Reverse proxy with TLS:** caddy `Caddyfile` example (auto-ACME), nginx config example.
4. **systemd unit example** for non-Docker self-host.
5. **Tailscale tailnet exposure** (bind 0.0.0.0, restrict via Tailscale ACLs).
6. **Token rotation procedure** (change env var, restart container; no revocation list since it's a single shared token).
7. **What to do if the token leaks** (rotate immediately; treat like a password).

### 4.8 Token model — explicit limits

A single shared bearer token = single user. There is no per-token identity, no audit trail per principal, no token revocation list. Multi-user is a different design and is out of scope. The deployment guide opens with this call-out so operators don't expect more than the model provides.

## 5. Batch `get_zim_entries`

### 5.1 Signature

Registered in [tools/content_tools.py](../../../openzim_mcp/tools/content_tools.py):

```python
@server.mcp.tool()
async def get_zim_entries(
    entries: list[dict[str, str]],   # [{"zim_file_path": "...", "entry_path": "..."}, ...]
    max_content_length: int = 5000,  # applied per-entry
) -> str:
    """Fetch multiple ZIM entries in one call.

    Returns JSON:
      {
        "results": [
          {"index": 0, "zim_file_path": "...", "entry_path": "...",
           "success": true, "title": "...", "content": "..."},
          {"index": 1, "zim_file_path": "...", "entry_path": "...",
           "success": false, "error": "Entry not found: ..."},
          ...
        ],
        "succeeded": int,
        "failed": int,
      }
    """
```

### 5.2 Partial-success semantics

Each entry resolves independently. One failure (entry not found, archive error, validation error) records `success: false` with the error message; remaining entries continue. The batch as a whole succeeds (non-error MCP result) as long as input parsed; per-item failures are data, not exceptions.

If the input list itself is invalid (empty, oversized, malformed), the entire call raises `OpenZimMcpValidationError` — boundary errors stay errors.

### 5.3 Limits

- `MAX_BATCH_SIZE = 50` (new constant in [constants.py](../../../openzim_mcp/constants.py)). Empty list → `OpenZimMcpValidationError`. Over 50 → `OpenZimMcpValidationError` with message "split into multiple batches".
- `max_content_length` validated against existing floor (`100` per 0.9.0 fix) and ceiling (`1_000_000`) once, applied to each entry.
- Different `zim_file_path` values within one batch are explicitly allowed (multi-archive workflow).

### 5.4 Caching

No special-cased batch cache. Each entry routes through the existing `OpenZimMcpCache.get` path, so duplicates within a batch deduplicate automatically by cache hits. Cache stats record N hits, not 1.

### 5.5 Rate limiting

`check_rate_limit("get_zim_entries")` called **once per entry**, not once per batch — otherwise batching becomes a rate-limit-bypass vector. If the rate limit trips mid-batch, remaining entries return `{"success": false, "error": "rate_limited"}`; completed work is preserved.

### 5.6 Async / sync split

Standard pattern:

1. Sync `ZimOperations.get_entries(...)` in [zim_operations.py](../../../openzim_mcp/zim_operations.py).
2. Async wrapper in [async_operations.py](../../../openzim_mcp/async_operations.py) using `asyncio.to_thread`.
3. MCP registration in [tools/content_tools.py](../../../openzim_mcp/tools/content_tools.py).

Internal loop is sequential — libzim is a synchronous C extension; intra-process parallelism buys nothing. Savings come from collapsed protocol round-trips, not parallel reads.

### 5.7 Validation

Each `entry` dict passes through `sanitize_input` + `path_validator.validate_path` + `validate_zim_file` at the boundary. Validation errors are recorded per-item (don't abort the batch).

### 5.8 Simple-mode intent pattern

Add to [simple_tools.py](../../../openzim_mcp/simple_tools.py):

| Trigger | Routes to | Specificity |
|---|---|---|
| `get entries X, Y, Z`, `fetch articles X and Y`, `read multiple X Y` | `get_zim_entries` | medium |

Pattern slots in by specificity so it doesn't shadow the existing `get_zim_entry` pattern.

## 6. Per-entry resources

### 6.1 Day-1 spike

A 1-day timeboxed investigation in [tools/resource_tools.py](../../../openzim_mcp/tools/resource_tools.py). Tested in this order, picking the first that works in current `mcp[cli]>=1.23.0`:

1. **`zim://{name}/entry/{+path}` or whichever syntax the SDK exposes for RFC 6570 reserved expansion.** Reserved expansion allows `/`; cleanest URL form.
2. **`zim://{name}/entry/{path}`** — plain capture; check if the SDK regex now permits `/` inside the captured segment.
3. **`zim://{name}/entry?path=A/Article`** — query-param fallback. Ugly but bulletproof; works regardless of URI-template support.
4. **None work →** punt the entire feature to 1.1; ship 1.0 with the other three items. Document `get_zim_entry` tool as the workaround.

The spike output is one of {form 1, form 2, form 3, "punted"}. Hard fork in the implementation plan; everything below assumes one of forms 1–3 succeeded.

### 6.2 Resource shape — native MIME

Deliberate departure from existing `zim://files` and `zim://{name}` resources (which return JSON because they're index/summary resources with no natural single-content type). Per-entry resources serve real content:

| Entry kind | `mimeType` | content |
|---|---|---|
| HTML article | `text/html` | raw HTML, **not** stripped to text |
| Plain text | `text/plain` | raw text |
| PDF | `application/pdf` | base64 blob |
| Image (png/jpg/webp/svg) | `image/*` | base64 blob |
| Other binary | `application/octet-stream` | base64 blob |

Rationale: resources are meant to *be* the thing, not summarize it. Tools (`get_zim_entry`) keep their text-stripping + truncation behavior for agents that want processed content. Resources serve raw bytes for clients that can render. This separation matches mature MCP servers in the ecosystem.

### 6.3 No truncation on resources

A 5 MB HTML article is returned in full. Truncating logic remains on `get_zim_entry`. If size becomes a real problem in practice, can add a resource-side `max_content_length` later — not premature.

### 6.4 Implementation

- Lives in [tools/resource_tools.py](../../../openzim_mcp/tools/resource_tools.py) next to existing two resources.
- Reuses `ZimOperations.get_entry()` internals — same path validation, same archive-error handling, same security pipeline.
- Resolves `{name}` → file path via the existing helper used by `zim://{name}`.
- New helper `_detect_mime_type(item)` consults libzim's `Item.mimetype` and falls back to `application/octet-stream` for unknown types.

### 6.5 Out of scope

- **Per-entry subscriptions.** ZIM entries are immutable per file; if the parent ZIM is replaced, the file-level subscription on `zim://{name}` covers it.
- **Range requests / partial content.** No client demand and adds non-trivial complexity.
- **Compression negotiation.** Reverse proxy handles gzip/brotli.

### 6.6 Risk and rollback

If the spike succeeds (forms 1 or 2) but post-spike implementation hits unforeseen issues, fall back to form 3 (query-param) without touching the rest of the release. Form 3 is uglier but functionally equivalent.

## 7. Resource subscriptions

### 7.1 Protocol

Standard MCP resource subscriptions:

1. Client → `resources/subscribe {uri}`.
2. Server records subscription against the session.
3. Server sends `notifications/resources/updated {uri}` when the source changes.
4. Client re-fetches via `resources/read`.
5. Client → `resources/unsubscribe {uri}` to stop, or session end auto-cleans.

### 7.2 Subscription targets

- **`zim://files`** — fires when allowed-dir contents change (`*.zim` added, removed, renamed).
- **`zim://{name}`** — fires when that specific file's mtime changes (atomic replacement by a download).
- **Per-entry — deliberately excluded.** Covered transitively by file-level on `zim://{name}`.

### 7.3 Change-detection mechanism — mtime polling

Background `asyncio` task wakes every `OPENZIM_MCP_WATCH_INTERVAL` seconds (default `5`, validated `1–60`). Each tick:

1. `os.scandir()` over each allowed dir, filtered to `*.zim`.
2. Build snapshot `{path: mtime}`.
3. Diff against previous snapshot: added paths, removed paths, mtime-changed paths.
4. For each diff entry, emit appropriate event into the registry dispatch path.

**Why polling over watchdog/inotify:** 5 s latency is fine for "I dropped a new ZIM in" scenarios. Polling works identically across Docker bind mounts, network mounts, macOS, Linux, Windows where filesystem-event APIs each have failure modes. Cost is one `scandir` per N seconds — negligible.

### 7.4 Subscription registry

New module `openzim_mcp/subscriptions.py`:

```python
class SubscriptionRegistry:
    # session_id -> set[uri]
    _subs: dict[str, set[str]]
    _lock: asyncio.Lock

    async def subscribe(self, session_id: str, uri: str) -> None
    async def unsubscribe(self, session_id: str, uri: str) -> None
    async def subscribers_for(self, uri: str) -> list[str]
    async def clear_session(self, session_id: str) -> None
```

Pure in-memory, no persistence — session-scoped state by definition.

### 7.5 FastMCP integration — the biggest unknown

FastMCP's high-level decorator API may not expose subscription hooks cleanly; the lower-level `mcp.server.Server` class does. Implementation may need to access FastMCP's underlying `Server` instance (the exact escape-hatch attribute name needs to be confirmed by the spike — likely a private attribute on the `FastMCP` instance) to register `subscribe_resources` / `unsubscribe_resources` handlers and call `send_resource_updated`.

**A 0.5-day sub-spike on Day 1 of this section verifies the API surface.** If it requires deep monkey-patching or fragile private-internal access likely to break on minor SDK upgrades, that's a slip trigger.

### 7.6 Notification dispatch

1. Polling task detects change → emits `(uri, change_type)` event into a queue.
2. Dispatch loop (separate `asyncio` task) reads queue, calls `subscribers_for(uri)`.
3. For each `session_id`, server sends `notifications/resources/updated` via the SDK's per-session send method.
4. SDK handles wire-level delivery (stdio stream or HTTP streaming session).

### 7.7 Cross-transport

Both stdio and HTTP support subscriptions identically from the client's perspective. Stdio has exactly one session = one process; HTTP has many sessions multiplexed on one process. The registry abstracts both.

### 7.8 Lifecycle

- **Subscribe:** add to registry, idempotent.
- **Unsubscribe:** remove, idempotent.
- **Session teardown** (HTTP `DELETE /` or connection drop, stdio process exit): server fires `clear_session(session_id)`.
- **Server shutdown:** cancel polling task, drop registry.

### 7.9 Configuration

- `OPENZIM_MCP_WATCH_INTERVAL=5` (seconds; 1–60).
- `OPENZIM_MCP_SUBSCRIPTIONS_ENABLED=true` (kill switch — set false to disable the polling task entirely; subscribe calls then return success but never fire updates, which is spec-permitted).

### 7.10 Slip triggers (any one → ship 1.0 without subscriptions, retarget 1.1)

1. FastMCP subscription API requires deep private-internals access clearly going to break on minor SDK upgrades.
2. Race conditions between the polling task and session teardown produce flaky tests that resist fixing within 2 working days.
3. Total subscriptions scope creeps past ~5–7 working days while the other three items are still mid-flight.

When any trigger fires, the documented fallback in CHANGELOG / release notes is: "Resource subscriptions are planned for 1.1. The `zim://files` and `zim://{name}` resources are fully readable in 1.0; clients can poll `resources/read` themselves at whatever cadence they prefer."

### 7.11 Out of scope

- Per-entry subscriptions (deferred indefinitely).
- Native FS-event backends (watchdog/inotify) — open follow-up if polling latency becomes a complaint.
- Subscription persistence across server restarts.
- Diff-style notifications (MCP spec only defines "updated").

### 8.2 Existing test files extended

- [tests/test_server.py](../../../tests/test_server.py): register/dispatch path for the new resource and tool.

### 8.3 Manual smoke before tagging

1. `make test` clean, `make test-cov` ≥80%, `make lint`, `make security`.
2. `docker build` succeeds for both arches; `docker run` with `OPENZIM_MCP_AUTH_TOKEN` set responds correctly via MCP Inspector.
3. **Stdio mode unchanged** — connect via Claude Desktop / Code with no config change, all tools and resources work.
4. HTTP mode behind caddy with TLS — Inspector connects with `Authorization: Bearer …`, calls each new tool/resource, verifies responses.
5. Drop a new `.zim` into the allowed dir — within 5–10 s, subscribed `zim://files` clients receive an update notification.
6. Replace an existing `.zim` (atomic rename) — `zim://{name}` subscribers notified.

### 8.4 Intentionally not tested

- Performance benchmarks for HTTP overhead. Existing `pytest-benchmark` covers hot paths; we're not promising perf SLAs in 1.0.
- Multi-user load / token rotation under concurrent requests. Single-user release; out of scope.
- Native TLS. We don't ship it.
- Watchdog / native FS-event paths. We don't ship them.

## 10. Risks and mitigations

| Risk | Mitigation |
|---|---|
| FastMCP subscription API requires private-internals access | Day 1 sub-spike. If unstable, defer subscriptions to 1.1 (slip trigger documented). |
| Per-entry resource URI template still blocked in current SDK | Day 1 spike with 3-tier fallback (`{+path}` → `{path}` → query-param). Punt feature entirely if all three fail. |
| Polling watcher races with session teardown produce flaky tests | Strict `asyncio.Lock` on registry; session-teardown cancels in-flight notifications via task group. If still flaky after 2 days, slip trigger fires. |
| Multi-arch Docker build fails on ARM emulation | `docker/setup-qemu-action` + `docker/setup-buildx-action`. Fallback: ship amd64-only for 1.0; add arm64 in a 1.0.x patch. |
| Auth token leak (via image, logs, process listing) | Token only via env var (never config file or CLI arg). Auth-failure logs exclude attempted token. SecretStr in config object. CHANGELOG explicitly states leak procedure. |
| Operator commits `OPENZIM_MCP_AUTH_TOKEN` to a config repo | Deployment guide opens with token-generation snippet and "treat like a password" call-out. `.env.example` template ships no real token. |
| Stdio users break on upgrade due to a regression | Stdio is the default and gets a full test pass. `tests/test_main.py` "no flags = stdio" assertion guards against regressions. |
| Subscription polling load on large allowed-dirs (>10k files) | `os.scandir` is `O(N)` per tick; 10k files at 5 s interval = 2k stats/sec, well below filesystem limits. Documented as a soft limit in deployment guide; suggest `OPENZIM_MCP_WATCH_INTERVAL=30` for very large dirs. |

## 11. Architecture notes

### 11.1 Module placement

- One new top-level module: `openzim_mcp/subscriptions.py` (registry + polling task).
- All other new code drops into existing modules following the established pattern:
  - HTTP middleware/health/CORS: small additions to [server.py](../../../openzim_mcp/server.py) (currently 353 lines; expected growth ~150 lines).
  - CLI flags: [main.py](../../../openzim_mcp/main.py) (currently 118 lines; expected growth ~50 lines).
  - Config fields: [config.py](../../../openzim_mcp/config.py) (currently 157 lines; expected growth ~30 lines).
  - Batch tool: [tools/content_tools.py](../../../openzim_mcp/tools/content_tools.py).
  - Per-entry resource: [tools/resource_tools.py](../../../openzim_mcp/tools/resource_tools.py).
  - Instance tracker: [instance_tracker.py](../../../openzim_mcp/instance_tracker.py) (currently 416 lines; schema extension only, no behavioral rewrite).

If `server.py` grows past ~500 lines, consider extracting HTTP-specific middleware into `openzim_mcp/http_middleware.py` as a follow-up — not a 1.0 concern.

### 11.2 Validation and security

All new tools and resources accept user input through `sanitize_input` from [security.py](../../../openzim_mcp/security.py) at the MCP boundary, with the same per-input length limits already defined in [constants.py](../../../openzim_mcp/constants.py). Path arguments go through `path_validator.validate_path` and `validate_zim_file`.

### 11.3 Rate limiting

New rate-limit categories: `get_zim_entries` (per-entry, see 5.5). All other new endpoints reuse existing categories or aren't rate-limited (health endpoints, subscribe/unsubscribe are session-bookkeeping ops with negligible cost).
