# `zim_get_section` true raw-text path (v2.5 #18) — design

**Milestone:** v2.5.0a3 · **Issue:** #18 · **Date:** 2026-06-08

## Problem

`zim_get_section` accepts `compact: bool` (default `True`) for surface uniformity with `zim_query` / `zim_get`, but at v2.0 the parameter is a **no-op**: section bodies always ship in the bundle's compact rendering. The backing bundle is built with `compact=True` hardcoded ([`bundle.py:336`](../../openzim_mcp/bundle.py)) precisely so a section matches the table-collapsing placeholders that `zim_get(view="full")` callers see — but that means a caller passing `compact=False` cannot get the unrendered (full-fidelity) section body the way the other tools allow.

**Target.** Wire a true raw-text path so `compact=False` returns the unrendered section body, matching the rendering contract `zim_get(compact=False)` already honours. Schema-additive — no existing caller has to opt in (the default `True` preserves today's behaviour).

## Scope decisions (settled in brainstorming)

- **Parameterize the shared bundle, default `True`.** The render mode flows through the existing bundle machinery rather than a parallel render path, so there is one source of truth for "render section HTML to markdown + compute section offsets." Default `True` keeps every other bundle consumer byte-identical.
- **Mode in the cache key.** The two render modes are distinct cache entries; an in-memory cache makes the key-format change free.
- **Only the section path passes `compact=False`.** Summary / TOC / structure / synthesize keep the compact bundle.
- **`compact_budget` stays a documented no-op.** Issue #18 is only the raw-text path; the budget knob is out of scope.
- **No default flip.** Whether to flip `zim_get`/section defaults is the separate telemetry-driven a4 item.

## Architecture

The change threads one boolean down an existing call chain and parameterizes the render at the bottom.

### 1. Bundle — `openzim_mcp/bundle.py`

- `extract_entry_bundle(archive, entry_path, *, content_processor, compact: bool = True)` — the hardcoded `_render_soup_to_text(content_root, compact=True)` ([line 336](../../openzim_mcp/bundle.py)) becomes `compact=compact`. `_compute_section_offsets(rendered, headings)` then computes offsets against the mode-appropriate `rendered_markdown`, so a slice of `rendered_markdown[char_start:char_end]` is in the requested fidelity.
- `get_or_build_bundle(..., compact: bool = True)` — forwards `compact` to `extract_entry_bundle` and includes the mode in the cache key.
- `_bundle_cache_key(validated_path, entry_path, compact)` — append a mode token (`:compact` / `:raw`) so `compact=True` and `compact=False` bundles never collide. The token is the only key-shape change.

The non-HTML early-return branch ([bundle.py:296-313](../../openzim_mcp/bundle.py)) is mode-independent and unchanged.

### 2. Data layer — `openzim_mcp/zim/structure.py`

- `get_section_data(..., compact: bool = True)` and `_get_section_data(..., compact: bool = True)` gain the flag and forward it to `get_or_build_bundle`. No other behaviour changes; section selection / widening logic is untouched.

### 3. Async wrapper — `openzim_mcp/async_operations.py`

- `get_section_data(..., compact: bool = True)` gains the flag and forwards it to the sync layer via `asyncio.to_thread`.

### 4. Tool — `openzim_mcp/tools/zim_get_section.py`

- Forward the already-declared `compact` argument to `ops.get_section_data(..., compact=compact)` instead of dropping it. Update the module docstring / inline comment that currently documents `compact` as a no-op.

## Rendering-fidelity contract

`compact=False` must match what `zim_get(view="full", compact=False)` produces for the same content. The two relevant differences `_render_soup_to_text` makes between modes:

- **Tables.** Compact replaces oversized tables with `[Table N: M rows x P cols - pass compact=False to expand]`; non-compact keeps the full pipe-delimited markdown. This is the primary user-visible win and the explicit reason the bundle hardcoded compact.
- **Infobox.** Compact extracts the infobox into KV markdown; non-compact does not.

One known asymmetry to verify and reconcile during implementation: `extract_entry_bundle` calls `_extract_infobox(content_root, …)` ([bundle.py:328](../../openzim_mcp/bundle.py)) which `decompose()`s the infobox from `content_root` **before** rendering, in *both* modes — whereas the direct `zim_get` full path only extracts the infobox when `compact=True`. The implementation must confirm whether `compact=False` section output and `zim_get(view="full", compact=False)` agree on infobox-bearing content (typically only the lead section), and reconcile if they diverge so the contract holds. The acceptance test below pins this.

## Testing strategy (TDD)

1. **Raw tables survive.** On an article with an oversized table, `compact=False` section output contains pipe-delimited table markdown; `compact=True` contains the `[Table N: …]` placeholder. (Red first: today both return the placeholder.)
2. **Compact unchanged.** `compact=True` (and the default call) returns byte-identical output to the pre-change behaviour for a representative section.
3. **Cache isolation.** Fetching the same section `compact=True` then `compact=False` returns different bodies — i.e. the second call is not served the first's cached compact bundle (mode is in the key).
4. **Tool forwards the flag.** `tests/test_zim_get_section.py` dispatch assertion updated so `get_section_data` is called with `compact`.
5. **Parity spot-check.** For a section, `compact=False` output is consistent with `zim_get(view="full", compact=False)` fidelity on the same content (tables expanded; infobox handling reconciled per the contract above).

## Backward compatibility & non-goals

- **Default `True`** ⇒ no behaviour change for any existing caller or the 4 non-section bundle consumers.
- **No schema change** (`GetSectionResponse.content_markdown` unchanged), **no `*_description.md` change** ⇒ the D14b prototype-parity guard does not fire and no re-snapshot is needed.
- `compact_budget` remains a no-op; default flips are deferred to a4.
