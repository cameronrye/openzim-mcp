# Docs Consolidation Design — Post-v2.0.0

**Date**: 2026-05-27
**Status**: Draft (pending user review)
**Driver**: v2.0.0 shipped 2026-05-27. README, GitHub Wiki, and the marketing website have drifted apart through Phase F. Three artifacts duplicate Install / Quick Start / Examples; the Wiki still shows the v1 22-tool API on most pages; the README has grown to 1,806 lines with seven stacked `What's new in ...` sections going back to v0.9.0; the landing page's "v1" section frames v1.0 as the latest release.

## Goal

Replace the README ⟷ Wiki ⟷ marketing-page tri-source with a single canonical documentation site backed by Astro, served at `https://cameronrye.github.io/openzim-mcp/`. README becomes a project card. Wiki is retired. The marketing landing page is preserved (bespoke design intact) and refreshed for v2.

## Non-goals

- No version switcher in docs — single-version site for v2.0.0 only.
- No search UI in the first cut.
- No CHANGELOG restructuring.
- No edits to `docs/roadmap.md` or `docs/extras-reranker.md` (internal planning).
- No new MCP discovery files in `.well-known/`.
- No edits to `llms.txt` content (just relocate to `public/`).

## Architecture

One Astro site at the existing `website/` folder. Bespoke landing page at `/`. Docs at `/docs/<slug>/`. One CI workflow deploys to `gh-pages`.

```
website/                                 # existing folder, becomes Astro project
├── astro.config.mjs                     # Astro + @astrojs/mdx + @astrojs/sitemap
├── package.json                         # astro, @astrojs/mdx, @astrojs/sitemap, shiki
├── tsconfig.json
├── src/
│   ├── pages/
│   │   ├── index.astro                  # bespoke landing (v1→v2 refresh applied)
│   │   └── docs/
│   │       └── [...slug].astro          # docs route, renders MDX
│   ├── content/
│   │   ├── config.ts                    # Zod schema: title, sidebar_order, group, summary
│   │   └── docs/                        # 15 MDX files (see Information Architecture)
│   ├── components/                      # ThemeToggle, CodeBlock, Callout, NavLink
│   ├── layouts/
│   │   ├── LandingLayout.astro
│   │   └── DocsLayout.astro             # sidebar + TOC + breadcrumbs + prev/next
│   └── styles/                          # current website/assets/*.css ports here
├── public/                              # robots.txt, humans.txt, llms.txt, .well-known/, assets/
└── dist/                                # build output, deployed to gh-pages
```

**Choice points:**

- **Astro, not Starlight.** Starlight's typography and layout opinions clash with the bespoke landing aesthetic. We hand-roll a ~200-line `DocsLayout.astro` (sidebar + TOC + breadcrumbs). Reversible if maintenance burden grows.
- **Site lives in `website/`**, not a new `docs-site/`. Preserves existing folder convention; keeps diff focused.
- **Sitemap is auto-generated** by `@astrojs/sitemap`. Hand-edited `website/sitemap.xml` is deleted.

**Deployment:**

- New workflow `.github/workflows/site.yml` triggers on push to `main` touching `website/**`.
- Steps: `npm ci` → `npm run build` (in `website/`) → `peaceiris/actions-gh-pages` publishes `website/dist/` to `gh-pages`.
- GitHub Pages config (source: `gh-pages` branch, root) is unchanged.
- Before PR 1 merges: tag `gh-pages` HEAD as `gh-pages-pre-astro` for instant rollback.

**URLs:**

| URL | Source |
| --- | --- |
| `/` | `src/pages/index.astro` (landing) |
| `/docs/` | `src/content/docs/index.mdx` (Introduction) |
| `/docs/<slug>/` | `src/content/docs/<slug>.mdx` (each docs page) |
| `/llms.txt` | `public/llms.txt` (unchanged content) |
| `/.well-known/*` | `public/.well-known/*` (unchanged content) |
| `/sitemap-index.xml` | Generated by `@astrojs/sitemap` |

| Worked examples (new) | n/a | Medium | New page from README Examples block; rewrite case studies for 8-tool surface. |

**Refresh standard, applied uniformly:**

1. v2 8-tool names in all examples. v1 names only appear in: API-reference migration table, FAQ ("v1 code, what changed?"), CHANGELOG links.
2. Default examples to simple mode (`zim_query`); add an advanced-mode sibling example only on pages about specific advanced tools.
3. Pick one notation per page: MCP JSON-RPC framing (`{"name": "...", "arguments": {...}}`) OR Python pseudo-call. No mixing within a page.
4. v1.x footer note (where load-bearing only): *"v1.x is in maintenance through 2026-11-27. See [CHANGELOG](.../CHANGELOG.md) for the v1→v2 migration table."*
5. Pin every page to v2.0.0. No "as of v1.x". No "in upcoming v2.5" forward-references except tied to a tracked issue (e.g., #199).

## Sequencing — 4 staged PRs

Each PR independently mergeable, revertible, with explicit exit gate. Order matters: scaffold first so docs links always resolve.

### PR 1 — Astro scaffold + landing port + CI deploy

**Scope:**

- New `website/` Astro project: `package.json`, `astro.config.mjs`, `tsconfig.json`. Add `node_modules/`, `dist/`, `.astro/` to `.gitignore`.
- Port `index.html` → `src/pages/index.astro` with the v1→v2 section refresh applied.
- Build thin `DocsLayout.astro` (~200 lines) + placeholder `src/content/docs/index.mdx` saying "Documentation is moving here from the GitHub wiki — coming in PR 2."
- New `.github/workflows/site.yml`: push-to-main on `website/**` → `npm ci && npm run build` → `peaceiris/actions-gh-pages` to `gh-pages`.
- Move `robots.txt`, `humans.txt`, `llms.txt`, `.well-known/`, `assets/` to `website/public/`.
- Delete `website/sitemap.xml`, `website/validate.py`.
- Pre-merge: tag current `gh-pages` HEAD as `gh-pages-pre-astro`.

**Exit gate:**

- `https://cameronrye.github.io/openzim-mcp/` loads with refreshed landing.
- `/docs/` renders the placeholder page.
- `/llms.txt`, `/.well-known/*`, `/robots.txt` still resolve at same URLs.
- `npm run build` and `astro check` clean.

### PR 2 — Migrate wiki content to docs collection

**Scope:**

- 15 MDX files in `src/content/docs/` per IA above (14 from wiki + 1 new `worked-examples.mdx` from README), applying the refresh standard.
- Sidebar nav wired from `content/config.ts` Zod schema.
- Merge `docs/deployment.md` → `src/content/docs/http-and-docker-deployment.mdx`. Delete `docs/deployment.md`.
- New "Release process" section appended to `CONTRIBUTING.md` (content from wiki `Release-System-Guide.md`).
- Cross-link: README, CONTRIBUTING, SECURITY, `llms.txt` no longer link to wiki URLs — all docs links go to `/docs/<slug>/`.

**Exit gate:**

- All 15 docs pages render.
- `npm run build` clean.
- `lychee` (or similar linkcheck) passes on `dist/`: no links resolve to `*/wiki/*` URLs from README, CONTRIBUTING, SECURITY, website, or llms.txt.

### PR 3 — Slim README to project card

**Scope:**

- Rewrite `README.md` from 1,806 → ~250 lines per the section list above.
- Drop all stacked `What's new in *` sections (content already in CHANGELOG).
- Verify Examples block content lives in PR 2's `worked-examples.mdx`; delete from README.
- Relocate Development / Testing / Monitoring / Versioning / Configuration / Security / Performance prose per the destination table.
- Trim badges to 2 rows.

**Exit gate:**

- README is 200–300 lines.
- Every README section either appears in the §"README shape" list or is removed.
- All README links resolve (linkcheck).

### PR 4 — Retire the GitHub Wiki

**Scope (this PR — stubs only; wiki feature stays enabled):**

- Clone wiki repo (`https://github.com/cameronrye/openzim-mcp.wiki.git`).
- Replace each of the 15 pages with a single-line redirect stub: `This page has moved → https://cameronrye.github.io/openzim-mcp/docs/<slug>/` plus a one-line context note.
- For `Release-System-Guide.md`: redirect to `https://github.com/cameronrye/openzim-mcp/blob/main/CONTRIBUTING.md#release-process`.
- Push to wiki `master`. Wiki feature stays enabled so the stub pages remain reachable.

**Exit gate:**

- Direct wiki URLs serve redirect stubs.
- New canonical URLs indexed via `sitemap-index.xml`.

**Follow-up chore (after ~30 days, not part of PR 4):**

- Once search engines have re-indexed the new canonical URLs (~30 days), disable the wiki feature: `gh api -X PATCH repos/cameronrye/openzim-mcp -f has_wiki=false`. Old URLs will then 404; acceptable since they are no longer indexed as primary results. Tracked separately so it does not block PR 4 review.

## Cross-PR invariants

- Every PR builds and passes existing CI standalone.
- `https://cameronrye.github.io/openzim-mcp/` keeps loading throughout — no maintenance window.
- `/llms.txt` stays at its current URL throughout (contract for LLM clients).
- llms.txt content remains canonically aligned with CHANGELOG.md (advisory only; not CI-enforced).

## Risks

| Risk | Mitigation |
| --- | --- |
| Old wiki URLs are indexed by search engines and external blogs; once wiki disabled, links 404 | PR 4 leaves redirect stubs in place for ~30 days before flipping `has_wiki=false`. Optional: keep wiki enabled indefinitely with stub-only pages (zero maintenance). |
| `peaceiris/actions-gh-pages` overwrites `gh-pages` each deploy; bad PR-1 build → site dark | Pre-PR-1 tag `gh-pages-pre-astro`. Rollback: `git push --force origin gh-pages-pre-astro:gh-pages` (under 60s). Documented in PR 1. |
| `astro check` + `npm run build` add a flaky step | Site workflow is decoupled from `test.yml` and `codeql.yml`. Docs build failure does not block test PRs. |
| Astro auto-sitemap differs from hand-curated priority/changefreq hints | Advisory hints; modern engines mostly ignore. Accept defaults. |
| External backlinks to deep README anchors (`#whats-new-in-v200`, `#api-reference`) break | PR 3 keeps `#whats-in-v200` anchor in the slimmed README. Other dropped anchors break; acceptable since backlinks should target docs site post-cutover. |
| llms.txt drift over time | Top-of-file note: *"Maintained alongside CHANGELOG.md; update both on each release."* Advisory only. |
